# Folder conventions вҖ” naming, frontmatter, lifecycle states

Mandatory conventions for the folder structure and files produced by `arch-brainstorm` (and processed by `arch-decision`, `sprint-manifest`).

## Directory structure

### During drafting (`docs/prd/`)

```
docs/prd/
в”ңв”Җв”Җ <area-slug>/                         вҶҗ area folder (kebab-case)
в”Ӯ   в”ңв”Җв”Җ 00-area-brief.md                 вҶҗ area index (mandatory)
в”Ӯ   в”ңв”Җв”Җ 01-<study-slug>.md               вҶҗ first study PRD
в”Ӯ   в”ңв”Җв”Җ 01-<study-slug>-diagrams.md      вҶҗ standalone diagrams for study 01
в”Ӯ   в”ңв”Җв”Җ 02-<study-slug>.md               вҶҗ second study
в”Ӯ   в”ңв”Җв”Җ 02-<study-slug>-diagrams.md
в”Ӯ   в””в”Җв”Җ audit/                           вҶҗ Quint FPF audit folders (created by arch-decision)
в”Ӯ       в”ңв”Җв”Җ 01-<study-slug>-fpf/         вҶҗ one folder per validated study
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ 00-context.md            вҶҗ Bounded Context (Phase 0)
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ 01-hypotheses.md         вҶҗ extracted hypotheses (Phase 1)
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ 02-verification.md       вҶҗ logical verification (Phase 2)
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ 03-validation.md         вҶҗ empirical evidence (Phase 3)
в”Ӯ       в”Ӯ   в”ңв”Җв”Җ 04-audit.md              вҶҗ R_eff via WLNK (Phase 4)
в”Ӯ       в”Ӯ   в””в”Җв”Җ 05-decision.md           вҶҗ global PRD verdict вҖ” DRR (Phase 5)
в”Ӯ       в””в”Җв”Җ 02-<study-slug>-fpf/
в”Ӯ           в””в”Җв”Җ вҖҰ(same 6 files)
```

### After promotion to a sprint

```
docs/sprints/Sprint-XX/
в”ңв”Җв”Җ sprint-manifest.md                   вҶҗ ONE aggregated manifest per sprint (all areas)
в””в”Җв”Җ <area-slug>/                         вҶҗ same area, same slug
    в”ңв”Җв”Җ 01-<study-slug>.md               вҶҗ PRDs moved from draft
    в”ңв”Җв”Җ 01-<study-slug>-diagrams.md
    в”ңв”Җв”Җ 02-<study-slug>.md
    в””в”Җв”Җ audit/
        в””в”Җв”Җ 01-<study-slug>-fpf/         вҶҗ FPF audit folders carried over
            в””в”Җв”Җ вҖҰ(00-context.md вҖҰ 05-decision.md)
```

One single `sprint-manifest.md` at the sprint root aggregates all PRDs into user stories + macro functional tasks with cross-area dependencies вҖ” not one manifest per PRD. Layout produced by the `sprint-manifest` skill.

## Naming convention

### Area slug

- **Kebab-case**, lowercase, no accents
- **Describes the domain**, not the specific feature
- **Stable** over time (does not change as the area evolves)

Good examples:
- `user-signup` вҖ” sign-up / onboarding domain
- `pdf-export` вҖ” PDF generation domain
- `notifications-push` вҖ” push notification domain
- `payment-reconciliation` вҖ” payment reconciliation domain

Bad examples:
- `user-signup-email-otp` вҖ” too specific (the OTP flow is a study inside the `user-signup` area)
- `fix-bug-123` вҖ” not an architectural area, it is a bug fix
- `refactor-backend` вҖ” too broad, "refactor" is not a domain

**Discriminator rule**: an area must reasonably be able to host multiple studies. If you are unsure whether something is "an area" or "a study", ask: *"Could there be another study inside this folder in the future?"* If yes, it is an area. If no, the study lives inside a larger area.

### Study slug

- **Kebab-case**, lowercase
- **Describes the specific aspect** studied
- **Distinct** from the other studies in the area
- Can contain qualifiers like version, alternative, sub-domain

Examples in the `user-signup` area:
- `email-otp` вҖ” study on the email OTP flow
- `passwordless-magic-link` вҖ” study on the magic-link flow
- `social-federation` вҖ” study on OIDC social providers

### File naming

Pattern:

```
NN-<study-slug>.md           вҶ’ the main PRD
NN-<study-slug>-diagrams.md  вҶ’ standalone Mermaid diagrams
audit/NN-<study-slug>-fpf/   вҶ’ FPF audit folder (generated by arch-decision; contains 00-context.md вҖҰ 05-decision.md)
```

Rules:
- `NN` = 2-digit numeric prefix, incremental in creation order (`01`, `02`, `03`, вҖҰ)
- Zero-padding is mandatory (`01`, not `1`) for alphanumeric ordering
- The main study and its diagrams share the same `NN-<study-slug>`

### Special cases

- **Area brief**: always `00-area-brief.md` (never renamed)
- **External markers**: argumentative folders external to the workflow have a `README-status.md` with `status: external` in the frontmatter

## YAML frontmatter

### PRD (file `NN-<study>.md`)

```yaml
---
title: "Readable PRD title"
area: <area-slug>
study: <study-slug>
status: draft | in-validation | validated | superseded | rejected | promoted
sprint: null | <number>
validated_at: null | YYYY-MM-DD
validated_by: null | "arch-decision (Quint FPF run <id>)"
superseded_by: null | "NN-<other-study>.md"
superseded_reason: null | "short reason text"
created_at: YYYY-MM-DD
authors: [name1, name2]
---
```

**All fields are mandatory**. `null` values mean "not applicable at the moment".

### Standalone diagrams (file `NN-<study>-diagrams.md`)

```yaml
---
title: "Diagrams вҖ” <study title>"
area: <area-slug>
study: <study-slug>
kind: diagrams
related_prd: ./NN-<study>.md
created_at: YYYY-MM-DD
---
```

### Area brief (file `00-area-brief.md`)

```yaml
---
title: "Area <area-slug> вҖ” study index"
area: <area-slug>
kind: area-brief
created_at: YYYY-MM-DD
---
```

### Audit folder (`audit/NN-<study>-fpf/`)

The audit is no longer a single file: it is a folder with one markdown per FPF phase. Each file carries its own frontmatter (defined by `quint-fpf` and `arch-decision`). The two cycle-level fields the rest of the workflow depends on are:

- `00-context.md` frontmatter `verdict_phase5` вҖ” empty while the cycle is open; set to `PASS | FAIL | NEEDS-REVISION` when `/q5-decide` (or `arch-decision` Phase 5) closes it.
- `05-decision.md` frontmatter `verdict` and `r_eff_min` вҖ” the canonical PRD-level result consumed by `sprint-manifest`.

Example `00-context.md` frontmatter:

```yaml
---
phase: 0
slug: <area>-<study>
output_dir: docs/prd/<area>/audit/NN-<study>-fpf/
mode: tooled | conversational
related_prd: ../../NN-<study>.md
problem_statement: <one-sentence summary>
created_at: YYYY-MM-DDTHH:MM:SSZ
verdict_phase5:
---
```

Example `05-decision.md` frontmatter (the DRR вҖ” final artifact):

```yaml
---
phase: 5
slug: <same as 00-context.md>
output_dir: <same>
mode: tooled | conversational
related_prd: ../../NN-<study>.md
decided_at: YYYY-MM-DDTHH:MM:SSZ
verdict: PASS | FAIL | NEEDS-REVISION
r_eff_min: 0.00
r_eff_avg: 0.00
---
```

## PRD lifecycle states

### State table

| State | Meaning | Who sets it | Possible next states |
|---|---|---|---|
| `draft` | Being written/reviewed, not yet validated | `arch-brainstorm` on creation | `in-validation`, `rejected` (manual) |
| `in-validation` | Quint FPF validation in progress | `arch-decision` at run start | `validated`, `rejected` |
| `validated` | Passed Quint FPF, ready for promotion | `arch-decision` at end of PASS run | `superseded`, `promoted`, `rejected` (manual, rare) |
| `superseded` | Was valid, but another study was chosen instead | Architect manually before `sprint-manifest` | terminal вҖ” no further transitions |
| `rejected` | Did not pass validation (FPF FAIL or manual choice) | `arch-decision` (FAIL) or architect | terminal |
| `promoted` | Moved into a sprint by `sprint-manifest` | `sprint-manifest` | terminal (lives now in sprint folder) |

### Difference between `rejected` and `superseded`

This distinction matters to preserve audit trail:

| Aspect | `rejected` | `superseded` |
|---|---|---|
| Passed Quint FPF? | No (FAIL) or manual discard | Yes (PASS) |
| Why was it discarded? | Technically invalid | Was valid, but another was preferred |
| `superseded_by` field | `null` | mandatory (points to the winner) |
| `superseded_reason` field | `null` | mandatory (short text) |
| Historical usefulness | Low (failed attempt) | High (considered alternative) |

**Superseded example** вҖ” area `api-gateway` with two studies:

```yaml
# 01-option-a.md
status: superseded
superseded_by: 02-option-b.md
superseded_reason: "Option B chosen for native integration with <platform>"
```

```yaml
# 02-option-b.md
status: validated
```

Both files stay in the folder. `sprint-manifest` ignores `01-option-a.md` but the file remains as a record of "we also considered option A and it passed Quint FPF, but option B was better becauseвҖҰ".

## Interaction with other skills

### `arch-brainstorm` (this skill) produces:
- Creates `<area>/` if it does not exist
- Creates `00-area-brief.md`
- Creates `NN-<study>.md` with `status: draft`
- Creates `NN-<study>-diagrams.md`
- Updates `00-area-brief.md` with the new study
- **Does not create** `audit/` (that is `arch-decision`'s job)

### `arch-decision` (next step):
- **Reads** `<area>/NN-<study>.md`
- Delegates the methodology to `quint-fpf`
- Updates frontmatter (`status: in-validation` вҶ’ `validated` or `rejected`)
- Creates `audit/NN-<study>-fpf/` with the 6 per-phase files (`00-context.md` вҖҰ `05-decision.md`)
- On PASS: produces a concrete "Impact on `docs/architecture/`" checklist for the architect to apply manually
- **Does not move** files (everything stays in `docs/prd/`)
- **Does not write** to `docs/architecture/` вҖ” that is the architect's responsibility

### Updating `docs/architecture/` (manual, after arch-decision PASS):
- The architect applies the "Impact on docs/architecture/" checklist produced by `arch-decision` Phase 6
- Typical edits: add a new component to `04-components.md`, add a constraint to `01-constraints.md`, append a new pattern to `03-patterns.md`
- Keep this change as a separate commit from the PRD validation вҖ” audit and source-of-truth changes stay decoupled

### `sprint-manifest` (when ready for sprint):
- **Scans** `<area>/` for all PRDs
- Requires all to be in `validated` state (`superseded` is ignored, `rejected` is ignored)
- Asks for target sprint (mandatory)
- **Moves** validated PRDs (+ their diagrams and audit reports) to `docs/sprints/Sprint-X/<area>/`
- Generates (or updates via impact analysis) **one aggregated manifest** at `docs/sprints/Sprint-X/sprint-manifest.md` вҖ” user stories, macro tasks, cross-area dependencies, CL1 risk flags
- Updates PRD frontmatter вҶ’ `status: promoted, sprint: X, promoted_at: YYYY-MM-DD`
- Updates `00-area-brief.md` with note "promoted to Sprint X"

## Special folders to ignore

### External argumentative folders

Folders that hold argumentative/reference documents **outside** the official workflow are marked with a `README-status.md` file:

```yaml
---
kind: area-status
status: external
---
```

Rules:
- `arch-brainstorm` **does not** touch these folders
- `arch-decision` **does not** validate documents inside them
- `sprint-manifest` **does not** promote them

If an architect wants to convert an external folder into a normal study area, they must:
1. Create a new folder with a different slug (`<new-slug>/`)
2. Use `arch-brainstorm` normally on the new folder
3. Optionally: cite the external document as "historical context" in the decision story
