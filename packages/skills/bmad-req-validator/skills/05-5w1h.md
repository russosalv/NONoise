# 5W1H — Requirement Information Completeness

## Origin and Context

The 5W (and 1H) is an information-elicitation technique rooted in 19th-century
investigative journalism and codified in the Kipling model ("I Keep Six Honest
Serving Men"). In IT and Requirements Engineering, the framework was formalized
as the "5W1H Requirement Cube" (IGI Global, 2021) and applied to connect business,
user and system requirements in a structured Enterprise Business Architecture
hierarchy.

## The Six Dimensions

### WHO
Identifies all actors involved in the requirement: who uses it, who triggers it,
who benefits, who has approval responsibility.

**Actor types:**
- **Direct users:** who interacts with the feature (e.g. customer, operator, admin)
- **External systems:** services, APIs, applications that interact
- **Indirect actors:** who benefits from the result without direct interaction
- **Data owners:** who owns and maintains the involved data

**Flag:** `[FAIL]` if the actor is generic ("the user", "the system") without
precise role identification.

---

### WHAT
Describes the specific action, feature or behavior required. Must be a concrete
action verb, not a generic one.

**Precise vs. vague actions:**
| Vague | Precise |
|---|---|
| manage | create, modify, delete, approve |
| support | compute, validate, send, generate |
| improve | reduce from X to Y, increase from A to B |

**Flag:** `[FAIL]` if the action is a generic verb that does not describe a
specific system behavior.

---

### WHERE
Specifies the application context, component, screen or service in which the
requirement applies.

**Dimensions of "where":**
- Application component (e.g. "in the checkout module", "in the authentication API")
- Environment (e.g. "in production", "cloud environment only")
- Geographic or legal location (e.g. "for EU users", "in the Italian datacenter")
- Interface (e.g. "in the mobile app", "in the web portal", "in the batch service")

**Flag:** `[WARN]` if the context is omitted but inferable; `[FAIL]` if ambiguous.

---

### WHEN
Defines the temporal conditions or events that activate the requirement.

**Types of "when":**
- **User event:** "when the user clicks on…", "when the file is uploaded"
- **System event:** "every 5 minutes", "on transaction completion"
- **State condition:** "when the balance is negative", "after authentication"
- **Business deadline:** "before the contract expiration date"

**Flag:** `[FAIL]` if no trigger or temporal context can be identified.

---

### WHY
Explains the business rationale or value the requirement delivers. This is
the most often neglected dimension but the most important for priority and
design decisions.

**Types of "why":**
- **User value:** time savings, error reduction, better experience
- **Business value:** cost reduction, revenue increase, regulatory compliance
- **Project objective:** alignment with KPI, contractual milestone, SLA
- **Regulatory constraint:** GDPR, sector regulations, audit requirements

**Flag:** `[FAIL]` if no motivation or value is stated; this is the most
critical gap for project governance.

---

### HOW
Describes the method, process or technical constraints with which the requirement
must be implemented. **Note:** specifying "how" is optional for high-level
functional requirements but mandatory for technical requirements and those
with integration constraints.

**When "how" is mandatory:**
- Integration constraints with external systems
- Technical standards to comply with (e.g. OAuth2, REST, SOAP)
- Performance is a requirement (e.g. "via Redis cache")
- Specific security requirements (e.g. "via HTTPS with TLS 1.3")

**Flag:** `[WARN]` if missing in a requirement with clear technical constraints;
`[PASS]` if missing in a pure functional requirement ("how" left to the team).

---

## 5W1H Validation Matrix

```
[ ] WHO:  Actors identified with specific role
[ ] WHAT: Action/behavior precisely described
[ ] WHERE: Application context defined
[ ] WHEN: Trigger or temporal condition present
[ ] WHY:  Business rationale/value stated
[ ] HOW:  Technical constraints (if applicable)
```

| Score | Status | Action |
|---|---|---|
| 6/6 (or 5/6 with HOW optional) | ✅ PASS | Information complete |
| 4-5/6 | 🟡 WARN | Refine missing dimensions |
| < 4/6 | 🔴 FAIL | Incomplete requirement — rewrite |

## Challenge Questions Template

| Dimension | Challenge Question |
|---|---|
| WHO missing | "Who are the users or systems that will use this feature? Do they have different roles or permissions?" |
| WHAT vague | "Can you describe more precisely what the system must do? Which specific operations must it perform?" |
| WHERE missing | "Which part of the application or which component does this requirement apply to?" |
| WHEN missing | "When should this behavior activate? What is the triggering event or condition?" |
| WHY missing | "What is the business value of this requirement? Why is it important to implement it?" |
| HOW missing (with constraints) | "Are there technical constraints on how this must be implemented? (e.g. protocols, standards, integration systems)" |

## Example Application

**Original requirement:** "Implement the sales report."

| Dimension | Analysis | Status |
|---|---|---|
| WHO | Not specified | ❌ FAIL |
| WHAT | "sales report" too generic | ❌ FAIL |
| WHERE | Not specified (web? export? email?) | ❌ FAIL |
| WHEN | Not specified (real-time? scheduled?) | ❌ FAIL |
| WHY | Not specified | ❌ FAIL |
| HOW | Not applicable at this level | ✅ N/A |

**Challenge questions generated:**
1. Who must access this report? (Sales manager? Admin? Customer?)
2. What data should the report contain? (Volume, value, by product, by region?)
3. How should it be delivered? (Web screen, PDF, Excel, scheduled email?)
4. At what frequency or following which event should it be generated?
5. What is the business value? (Strategic decisions? Audit? Monthly KPIs?)
