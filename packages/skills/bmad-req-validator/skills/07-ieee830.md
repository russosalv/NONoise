# IEEE 830 / ISO 29148 — Requirement Quality

## Origin and Context

IEEE 830-1998 ("Recommended Practice for Software Requirements Specifications")
defines the content and quality attributes of a good SRS. It was later updated
and superseded by ISO/IEC/IEEE 29148:2018 ("Systems and software engineering —
Life cycle processes — Requirements engineering"), which extends the framework
to system and stakeholder requirements in addition to software ones.

These standards are fundamental in:
- Enterprise contexts in regulated sectors (banking, finance, healthcare, public sector)
- Tender/RFP projects where documentary quality is evaluated
- Audits and ISO/CMMI certifications
- Safety-critical projects

## The Eight Requirement Qualities (IEEE 830)

### 1. Correct
Each requirement must represent exactly what the software must do.

**Check:** has the requirement been validated by the business owner or
appropriate stakeholder? Does it accurately reflect their expectations?

**Flags:**
- `[PASS]` if the requirement has sign-off or was explicitly validated
- `[WARN]` if derived from unverified assumptions
- `[FAIL]` if it explicitly contradicts a stakeholder statement

---

### 2. Unambiguous
Each requirement must have a single possible interpretation. Every technical
term must be defined in a glossary.

**Anti-patterns to detect:**
- Pronouns without antecedent ("it should", "they must")
- Domain-overloaded terms
- Conditions combined with AND/OR without logical parentheses
- Comparatives without baseline ("faster than", "better than")
- Subjective terms: "easy", "user-friendly", "intuitive", "robust"

**Flags:**
- `[PASS]` if the requirement has a single plausible interpretation
- `[WARN]` if 2 plausible interpretations exist
- `[FAIL]` if it can be interpreted in substantially different ways

---

### 3. Complete
The requirement must cover all significant cases: valid AND invalid input,
normal AND exceptional behaviors.

**Check:**
- Are system responses defined for all relevant input types?
- Are both success and error paths handled?
- Are all references to standards or other documents included?

**Flags:**
- `[PASS]` if both happy path and error path are defined
- `[WARN]` if the error path is missing but the requirement is simple
- `[FAIL]` if the requirement is incomplete on critical aspects

---

### 4. Consistent
The requirement must not contradict other system requirements or project
documents (architecture, other SRS, contracts).

**Check:**
- Does the requirement use the same terminology as others?
- Does it contradict already-approved requirements?
- Is it consistent with documented architectural constraints?

**Flags:**
- `[PASS]` if no contradictions are detected
- `[WARN]` if a potential conflict must be verified
- `[FAIL]` if it explicitly contradicts an existing requirement or constraint

---

### 5. Ranked for Importance/Stability
Each requirement must have an assigned priority (e.g. MoSCoW) and an indication
of stability (stable, under discussion, subject to change).

**Check:**
- Does the requirement have an assigned priority?
- Is stability indicated?
- Is the priority consistent with that of related requirements?

**Flags:**
- `[PASS]` if priority and stability are indicated
- `[WARN]` if priority is present but stability is missing
- `[FAIL]` if neither priority nor stability is present

---

### 6. Verifiable
It must be possible to demonstrate, via testing or inspection, that the
requirement is satisfied. A non-verifiable requirement cannot be accepted.

**Check:**
- Is there a practicable verification method (test, analysis, inspection, demo)?
- Do the acceptance criteria allow a binary answer (satisfied/not satisfied)?
- Are the required test data defined?

**Non-verifiable anti-patterns:**
- "The system must always be available" (999% uptime? How is it measured?)
- "The system must be secure" (against which threat model?)
- "The system must be maintainable" (metric? threshold?)

**Flags:**
- `[PASS]` if a verifiable acceptance criterion exists
- `[WARN]` if only qualitatively verifiable
- `[FAIL]` if no test can be written for this requirement

---

### 7. Modifiable
The SRS must be structured so requirement changes do not impact unrelated
requirements. At the single-requirement level: it must be atomic and non-redundant.

**Check:**
- Is the requirement atomic (describes a single behavior)?
- Does it not duplicate information already present elsewhere?
- Does it have a unique ID for traceability?

**Flags:**
- `[PASS]` if atomic with unique ID
- `[WARN]` if composite (multiple behaviors in one) but splittable
- `[FAIL]` if duplicates others or lacks an identifier

---

### 8. Traceable
Each requirement must have an origin (stakeholder, business goal, regulation)
and must be traceable forward to design, code and tests.

**Check:**
- Does the requirement have a unique ID?
- Is the source indicated (e.g. meeting minutes, contract, regulation)?
- Can it be linked to test cases and architectural components?

**Flags:**
- `[PASS]` if ID, source and forward traceability are present
- `[WARN]` if only the ID is present but source and traceability are missing
- `[FAIL]` if no ID and not traceable

---

## IEEE 830 Scoring Schema

```
Score: [Correct:?] [Unambiguous:?] [Complete:?] [Consistent:?]
       [Ranked:?] [Verifiable:?] [Modifiable:?] [Traceable:?]
Total: X/8
```

| Score | Status |
|---|---|
| 7-8/8 | ✅ PASS — Standard compliant |
| 5-6/8 | 🟡 WARN — Improvements needed |
| < 5/8 | 🔴 FAIL — Non-compliant |

## Challenge Questions Template

| Quality | Challenge Question |
|---|---|
| Correct | "Has this requirement been validated by the business owner? Is there formal sign-off?" |
| Unambiguous | "Could the term '[X]' mean different things to different people? Should we add it to the glossary?" |
| Complete | "How should the system behave if it receives invalid input or an error occurs?" |
| Consistent | "Does this requirement use the same terminology as the others? Are there possible conflicts?" |
| Ranked | "What is the priority of this requirement (Must/Should/Could)? Is it stable or might it change?" |
| Verifiable | "How will you demonstrate in testing that this requirement has been correctly implemented?" |
| Modifiable | "Does this requirement describe a single behavior or more than one?" |
| Traceable | "What is the unique ID of this requirement? From which document or request does it come?" |
