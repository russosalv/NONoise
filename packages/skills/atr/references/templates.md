# ATR Output Templates

Templates for reports, backlog entries, delegated agent prompts, and stakeholder-friendly test plans used across `/atr run` and `/atr doc`.

All snippets below use **English** as default display language; adapt to `meta.language` of the testbook when rendering.

## Delegated Agent Prompt

When delegating report writing to a background agent, provide all the data it needs. The delegated agent has no browser access — it only writes files.

```
Update the ATR report file at {report_path}.

Test result to append:
- ID: {test_id}
- Title: {test_title}
- Manifest ref: {manifest_ref}
- Status: {PASS|FAIL|SKIP}
- Preconditions: {preconditions}
- Evidence screenshot: {exact filename from testbook step, or {test_id}.png if none defined}
- Verifications: {list of check text + pass/fail + observation}
- Notes: {any notes}
- Skip reason: {if SKIP}

Instructions:
1. Read the current report file
2. Append the test-result block using the ATR report format (below)
3. Update the summary table at the top: increment the {status} count by 1
4. If status is FAIL, also read/create {backlog_path} and append a failure entry
```

## Report Format

Each section (one per area of the sprint) gets its own markdown file.

```markdown
# ATR Sprint {N} — {Section Title}

**Run date**: {date} {time}
**Runner**: ATR skill
**Manifest reference**: {manifest_ref}
**App URL**: {app_url}
**Auth**: {auth.type from testbook}

## Summary

| Status | Count |
|--------|-------|
| PASS   | {count} |
| FAIL   | {count} |
| SKIP   | {count} |

---

## {test_id} — {test_title}

**Manifest**: {manifest_ref} | **Status**: {PASS|FAIL|SKIP}

**Preconditions**: {preconditions joined}

**Screenshot**:
![{caption}](screenshots/{filename})

**Verifications**:
- [x] Verification that passed
- [ ] Verification that failed <- **FAIL: {what was observed instead}**

**Notes**: {any additional observations}

---
```

## Report Format (SKIP)

```markdown
## {test_id} — {test_title}

**Manifest**: {manifest_ref} | **Status**: SKIP

**Reason**: {skip_reason}

**Requirements to verify** (when the feature becomes available):
- [ ] {expected item 1}
- [ ] {expected item 2}
```

## Backlog Format

```markdown
# ATR Sprint {N} — Failure Backlog

**Date**: {date}
**Total failures**: {count}
**Testbook**: {testbook path}

This document lists every test that failed during ATR execution.
Each entry has enough context to be worked on autonomously by a separate AI session.

---

## {test_id} — {short description of the failure}

- **Manifest section**: {manifest_ref}
- **Test title**: {test_title}
- **Manifest requirement**: "{exact requirement text from sprint manifest}"
- **Expected**: {expected}
- **Observed**: {actual observation}
- **Screenshot**: [screenshots/{filename}](screenshots/{filename})
- **Severity**: {critical|high|medium|low}
- **Likely files**: `{path to suspected source files}`
- **Fix hint**: {AI's best guess at what needs to change}
```

---

## Severity Guide

| Severity | Criteria |
|----------|----------|
| **critical** | Feature completely broken; blocks core workflow |
| **high** | Feature works but a key manifest requirement is not met |
| **medium** | Minor UI/UX mismatch (wrong label, missing hyphen, styling off) |
| **low** | Edge case not handled; cosmetic issue |

## Stakeholder Test Plan Document

Used by `/atr doc` to generate a stakeholder-friendly document from a testbook YAML. Written in the project's working language (from `meta.language`), professional tone, no jargon.

### Document Structure

```markdown
# Test Plan — Sprint {N}

| | |
|---|---|
| **Project** | {project name, from nonoise.config.json} |
| **Sprint** | Sprint {N} |
| **Reference document** | {sprint manifest path} |
| **Generated** | {date} |
| **Generated by** | ATR (Acceptance Test Runner) |

## Introduction

This document describes the acceptance-test plan for Sprint {N}.
The tests are derived from the sprint manifest and verify that the application
meets the functional requirements agreed with stakeholders.

Each test case describes:
- **what** is being verified (the functional requirement)
- **how** it is verified (the actions performed on the application)
- **what result** is expected

Tests are executed automatically using browser-automation tooling,
with visual evidence (screenshots) captured at every verification step.

## Summary

| Section | Manifest ref | Test cases | Of which not verifiable |
|---------|--------------|------------|-------------------------|
| {section title} | {manifest_ref} | {count} | {skip count} |
| ... | ... | ... | ... |
| **Total** | | **{total}** | **{total skip}** |

---

## {N}. {Section Title}

*Manifest reference: {manifest_ref}*

{Brief 1-2 sentence description of what this section covers, derived from the
sprint manifest's task descriptions. Example: "This section verifies the signup
flow via email and OTP, as described in tasks T1–T3 of the manifest."}

### {N}.{M} {Test Title}

**ID**: {test_id}
**Manifest reference**: {manifest_ref}

**Initial conditions**:
- {precondition 1, rephrased in business language}
- {precondition 2}

**Actions performed**:
1. {Step description in business language — combine multiple technical steps into one logical action}
2. {Next logical action}
3. Visual evidence is captured: "{screenshot caption}"
4. {Verification rephrased as "The system verifies that..."}

**Expected results**:
- {expected item 1}
- {expected item 2}

---
```

### For skipped tests

```markdown
### {N}.{M} {Test Title}

**ID**: {test_id}
**Manifest reference**: {manifest_ref}
**Status**: Not verifiable

**Reason**: {skip_reason rephrased in business language}

**Requirements to verify** (when the feature becomes available):
- {expected item 1}
- {expected item 2}

---
```

### Document Footer

```markdown
---

## Methodology notes

The tests described in this document are executed automatically using browser
automation that simulates real user actions against the application.

Each test case captures at least one piece of visual evidence (a screenshot)
documenting the state of the application at the verification moment.
These pieces of evidence are attached to the execution report.

Tests marked as "Not verifiable" concern features not yet implemented or
requiring special conditions not reproducible in the test environment.

### Status legend

| Status | Meaning |
|--------|---------|
| Verifiable | The test can be executed automatically |
| Not verifiable | The test requires features not yet implemented or special conditions |

### ID prefix legend

| Prefix | Functional area |
|--------|-----------------|
| {PR1-} | {Area 1 from sprint manifest} |
| {PR2-} | {Area 2 from sprint manifest} |
```

## Transformation Examples

These show how to convert testbook YAML steps into stakeholder language.

**Example 1 — Navigation + table verification:**

```yaml
steps:
  - action: goto
    target: "/users"
  - action: wait
    duration: 3
  - action: snapshot
    verify:
      - "User table visible with data rows"
      - "'Name' column present"
      - "'Status' column present"
```

Becomes:

> 1. The user navigates to the Users section
> 2. The system verifies that:
>    - The user table is visible and contains data
>    - The columns "Name" and "Status" are present

**Example 2 — Form filling:**

```yaml
steps:
  - action: fill
    description: "Enter First Name"
    target_hint: "'First name' field in the personal-data form"
    value: "Alice"
  - action: fill
    description: "Enter Last Name"
    target_hint: "'Last name' field in the personal-data form"
    value: "Smith"
  - action: select
    description: "Select country"
    target_hint: "'Country' dropdown in the personal-data form"
    value: "IT"
```

Becomes:

> 1. The user fills the form with the following data:
>    - First name: "Alice"
>    - Last name: "Smith"
>    - Country: "IT"

**Example 3 — Console error capture (technical):**

```yaml
steps:
  - action: eval
    script: "window.__ATR_CONSOLE_ERRORS = []; ..."
    expected: "Console interceptor installed"
```

Becomes:

> 1. The system activates application-error monitoring for the duration of the test

**Example 4 — Data verification after creation:**

```yaml
steps:
  - action: eval
    script: >
      JSON.stringify({
        hasFirstName: document.body.textContent.includes('Alice'),
        hasLastName: document.body.textContent.includes('Smith')
      })
    expected: "contains hasFirstName:true, hasLastName:true"
```

Becomes:

> 1. The system verifies that the entered data is displayed correctly:
>    - First name "Alice" present in the record
>    - Last name "Smith" present in the record
